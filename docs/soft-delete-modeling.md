## Soft Delete Modeling: The "Safety Net" for Your Data

> ডেটাবেজ থেকে ডেটা পার্মানেন্টলি ডিলিট করা মানে হলো অনেকটা নিজের স্মৃতিশক্তি হারিয়ে ফেলার মতো। প্রোডাকশন লেভেলে ভুল করে একটা DELETE কুয়েরি চালানো মানে পুরো কোম্পানির অবস্থা খারাপ করে দেওয়া। এই রিস্ক কমানোর জন্যই Soft Delete ব্যবহার করা হয়।

সফটওয়্যার ইঞ্জিনিয়ারিংয়ে আমরা প্রায়ই এমন পরিস্থিতির সম্মুখীন হই যেখানে ইউজার ভুল করে কিছু ডিলিট করে ফেলে এবং পরে তা ফিরে পেতে চায়। Soft Delete হলো এমন একটি টেকনিক যেখানে আমরা ডেটাবেজ থেকে ডেটা আসলে ডিলিট করি না, বরং সেটিকে "ডিলিটেড" হিসেবে মার্ক করে রাখি।

### ১. Overview (সহজ ভাষায়)

- **Hard Delete**: ডেটাবেজ থেকে রেকর্ডটি চিরতরে মুছে ফেলা হয় `(DELETE FROM table WHERE id = 1)`. এটি একবার করলে আর ফিরে পাওয়া সম্ভব নয়।
- **Soft Delete**: রেকর্ডটি টেবিলেই থাকে, শুধু একটি ফ্ল্যাগ (Flag) ব্যবহার করে সেটিকে হাইড (Hide) করে রাখা হয়। এটি অনেকটা কম্পিউটারের Recycle Bin-এর মতো কাজ করে।

### ২. Implementation (কীভাবে করবেন?)

বেস্ট প্র্যাকটিস হলো `is_deleted` (Boolean) এর বদলে `deleted_at` (Timestamp) কলাম ব্যবহার করা। কারণ এতে আপনি জানতে পারেন ডেটাটি কখন ডিলিট করা হয়েছিল।

```sql
ALTER TABLE users ADD COLUMN deleted_at TIMESTAMP NULL;
```

এখন কোনো ইউজারকে ডিলিট করতে চাইলে আমরা এটি করব:

```sql
UPDATE users SET deleted_at = NOW() WHERE id = 101;
```

আর ডেটা রিড করার সময় সবসময় ফিল্টার ব্যবহার করতে হবে:

```sql
SELECT * FROM users WHERE deleted_at IS NULL;
```

### ৩. Unique Constraint Trap

শেখার শুরুতে অনেকেই একটি বড় ভুল করে। ধরুন আপনার users টেবিলে email কলামটি UNIQUE।

১. ইউজার nasim@example.com তার অ্যাকাউন্ট ডিলিট করল (Soft Delete)।

২. এখন তার deleted_at কলামে একটি টাইমস্ট্যাম্প আছে।

৩. নতুন একজন ইউজার আবার একই nasim@example.com ইমেইল দিয়ে রেজিস্ট্রেশন করতে চাইল।

৪. Boom! ডেটাবেজ এরর দিবে কারণ ওই ইমেইলটি অলরেডি টেবিলে আছে (যদিও সেটি ডিলিটেড)।

**Solution:**

- **Partial Indexes (PostgreSQL):** শুধু ওই রেকর্ডগুলোর ওপর ইউনিক ইনডেক্স করা যারা ডিলিট হয়নি।

```sql
CREATE UNIQUE INDEX idx_user_email ON users(email) WHERE deleted_at IS NULL;
```

- **Composite Unique Key:** ইমেইল এবং `deleted_at` কলাম দুটোকে মিলিয়ে ইউনিক কি বানানো।

### ৪. At a Glance (সুবিধা অসুবিধা)

| সুবিধা (Pros)                                                     | সীমাবদ্ধতা (Cons)                                                                         |
| ----------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| Accidental Recovery: ডিলিট হওয়া ডেটা ১ সেকেন্ডে ফিরিয়ে আনা সম্ভব। | Storage Cost: ডেটা আসলে ডিলিট হয় না বলে স্টোরেজ বাড়তে থাকে।                               |
| Audit Log: ডিলিট করার ইতিহাস (কে কখন করেছে) রাখা যায়।             | Query Complexity: প্রতিবার WHERE deleted_at IS NULL লিখতে হয়, যা ভুল হওয়ার সম্ভাবনা থাকে। |
| Data Integrity: ফরেন কি (Foreign Key) এর ঝামেলা হয় না।            | Performance: টেবিল বড় হয়ে গেলে ইনডেক্সিং এবং কুয়েরি স্লো হতে পারে।                        |

### ৫. বাস্তব জীবনের উদাহরণ (Use Cases)

**১. SaaS Application:** সাবস্ক্রিপশন ক্যানসেল করলে ডেটা রেখে দেওয়া যাতে পরে রি-অ্যাক্টিভেট করা যায়।

**২. E-commerce:** ইনভেন্টরি বা অর্ডার হিস্টোরি ভুল ডিলিট হওয়া থেকে বাঁচাতে।

**৩. Social Media:** পোস্ট বা কমেন্ট ডিলিট করলে সেটি ব্যাকএন্ডে কিছুদিন জমা রাখা (যেমন- Instagram-এর "Recently Deleted" ফিচার)।

## নোটঃ

- **Global Scope:** লারাভেল (Laravel) বা জ্যাঙ্গো (Django) এর মতো ফ্রেমওয়ার্ক ব্যবহার করলে "Soft Deletes" ট্রেইট ব্যবহার করুন। এটি অটোমেটিক সব কুয়েরিতে ফিল্টার বসিয়ে দিবে।

- **GDPR Compliance:** মনে রাখবেন, ইউরোপীয় ইউনিয়ন বা অনেক দেশে "Right to be Forgotten" আইন আছে। তাই ইউজার যদি পার্মানেন্ট ডিলিট চায়, তবে আপনাকে অবশ্যই Hard Delete করতে হবে।

- **Cleanup Job:** স্টোরেজ বাঁচাতে একটি ব্যাকগ্রাউন্ড জব (Cron Job) রাখুন যা ৩০ বা ৬০ দিনের পুরনো সফট ডিলিটেড ডেটাগুলো পার্মানেন্টলি ক্লিন করবে।
