## Audit Table Design: The Database CCTV

কোডে বাগ (Bug) থাকা স্বাভাবিক, কিন্তু ডেটাবেজে কে, কখন, কী পরিবর্তন করেছে তা না জানাটা অপরাধ। প্রোডাকশন এনভায়রনমেন্টে যখন কোনো ডেটা রহস্যজনকভাবে বদলে যায়, তখন Audit Table-ই আমাদের একমাত্র উদ্ধারকর্তা।

> ডেটাবেজ ডিজাইনে Audit Table হলো এমন একটি ব্যবস্থা যা সিস্টেমের প্রতিটি পরিবর্তনের (Insert, Update, Delete) ইতিহাস রেকর্ড করে রাখে। এটি অনেকটা ব্যাংকের লেজার বা দোকানের ক্যাশ মেমোর মতো—যেখানে প্রতিটি লেনদেনের হিসাব থাকে।

### ১. Overview (সহজ ভাষায়)

সহজ কথায়, অডিট টেবিল হলো আপনার ডেটাবেজের জন্য একটি CCTV ক্যামেরা।

- **সাধারণ টেবিল:** শুধু বর্তমান অবস্থা (Current State) দেখায়।
- **অডিট টেবিল:** দেখায় "আগে কী ছিল" এবং "এখন কী হয়েছে"।

### ২. অডিট টেবিলের ৪টি "W" (The 4 W's)

টেবিল ডিজাইন করার সময় এই ৪টি প্রশ্নের উত্তর নিশ্চিত করাটা অনেক গুরুত্বপূর্ণ

- **Who**: পরিবর্তনটি কে করেছে? (User ID)
- **When**: কখন পরিবর্তনটি হয়েছে? (Timestamp)
- **What**: কোন টেবিলে এবং কোন রেকর্ডে পরিবর্তন হয়েছে? (Table Name, Record ID)
- **Where (Optional)**: কোন আইপি (IP) বা সোর্স থেকে পরিবর্তনটি হয়েছে?

### ৩. Implementation Strategies (ডিজাইন প্যাটার্ন)

অডিট টেবিল মূলত দুটি জনপ্রিয় পদ্ধতিতে ডিজাইন করা যায়:

#### পদ্ধতি ক: Single Centralized Audit Table (সবার জন্য এক টেবিল)

সব টেবিলের অডিট লগ একটি মাত্র টেবিলে রাখা হয়।

- **উপকারিতা**: মেইনটেইন করা সহজ।
- **কলামসমূহ**: id, table_name, record_id, action (Insert/Update/Delete), old_values (JSON), new_values (JSON), changed_by, created_at.

#### পদ্ধতি খ: Shadow Tables (টেবিল প্রতি আলাদা অডিট টেবিল)

প্রতিটি মেইন টেবিলের জন্য একটি আলাদা অডিট টেবিল (যেমন: users টেবিলের জন্য users_audit)।

- **উপকারিতা**: কুয়েরি করা এবং রিপোর্ট জেনারেট করা খুব ফাস্ট হয়।
- **সীমাবদ্ধতা**: অনেক বেশি টেবিল তৈরি হয়, যা মেইনটেইন করা কঠিন।

### ৪. SQL Implementation (JSON Approach)

```sql
CREATE TABLE audit_logs (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    table_name VARCHAR(50) NOT NULL,
    record_id INT NOT NULL,
    action ENUM('INSERT', 'UPDATE', 'DELETE') NOT NULL,
    old_values JSON NULL, -- পরিবর্তনের আগের ডেটা
    new_values JSON NULL, -- পরিবর্তনের পরের ডেটা
    user_id INT NULL,
    ip_address VARCHAR(45),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### ৫. At a Glance (সুবিধা অসুবিধা)

| সুবিধা                                                                         | সীমাবদ্ধতা                                                                                     |
| ------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------- |
| Accountability: কেউ কোনো ভুল করলে বা জালিয়াতি করলে সহজে ধরা যায়।               | Storage Overload: মূল টেবিলের চেয়ে অডিট টেবিল ১০ গুণ দ্রুত বড় হতে পারে।                        |
| Debugging: প্রোডাকশনে কোনো বাগ ডেটা নষ্ট করলে তার আগের ভার্সন খুঁজে পাওয়া যায়। | Performance Hit: প্রতিটি রাইট অপারেশনে এক্সট্রা একটি ইনসার্ট করতে হয় বলে কিছুটা স্লো হতে পারে। |
| Compliance: ফিনটেক বা হেলথ-টেক অ্যাপে এটি আইনি বাধ্যবাধকতা।                    | Complexity: ডিলিট হওয়া রেকর্ডের অডিট রাখা কিছুটা চ্যালেঞ্জিং।                                  |

## নোটঃ

- **Use Database Triggers**: যদি আপনি নিশ্চিত করতে চান যে কোনোভাবেই অডিট লগ মিস হবে না (এমনকি সরাসরি SQL রান করলেও), তবে Database Triggers ব্যবহার করুন।
- **JSON is Your Friend**: old_values এবং new_values কলামে পুরো রো (Row) স্টোর না করে শুধু যা পরিবর্তন (Diff) হয়েছে তা স্টোর করুন। এতে স্টোরেজ বাঁচবে।
- **Retention Policy**: অডিট ডেটা সারাজীবন রাখার দরকার নেই। একটি পলিসি রাখুন যে ৬ মাস বা ১ বছর পর পুরনো লগগুলো আর্কাইভ বা ডিলিট করে দেওয়া হবে।
- **Don't Audit Everything**: পাসওয়ার্ড বা সেনসিটিভ ডেটা (যেমন- Credit Card No) কখনোই অডিট টেবিলে ইনসার্ট করবেন না। এটি সিকিউরিটি রিস্ক।
