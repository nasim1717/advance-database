## Partial / Filtered Index: Precision Over Mass

> সবচেয়ে স্মার্ট ইনডেক্স সেটিই নয় যা সব ডেটাকে কভার করে, বরং সেটিই যা অপ্রয়োজনীয় ডেটাকে বাদ দিয়ে শুধু দরকারী অংশটুকু নিয়ে কাজ করে। ডেটাবেজ অপ্টিমাইজেশনের দুনিয়ায় একেই আমরা বলি Partial Index বা Filtered Index।

সাধারণত একটি ইনডেক্স টেবিলের প্রতিটি রো (Row)-এর জন্য এন্ট্রি তৈরি করে। কিন্তু বাস্তব জীবনে আমাদের সব ডেটার ওপর সার্চ করার প্রয়োজন হয় না। Partial Index হলো এমন একটি ইনডেক্স যা শুধুমাত্র একটি নির্দিষ্ট শর্ত (Where Clause) পূরণকারী ডেটাগুলোর ওপর তৈরি হয়।

### ১. Overview

সহজ কথায়, এটি একটি "স্মার্ট ফিল্টার"। আপনি ডেটাবেজকে বলছেন— "পুরো টেবিল ইনডেক্স করার দরকার নেই, শুধু সেই রেকর্ডগুলো ইনডেক্স করো যেগুলো আমার কুয়েরিতে সবচেয়ে বেশি ব্যবহৃত হয়।"

- **সহজ উদাহরণ:** আপনার একটি ই-কমার্স সাইট আছে যেখানে লাখ লাখ Orders আছে। কিন্তু আপনি সবসময় শুধু সেই অর্ডারগুলো খোঁজেন যেগুলো 'Pending' অবস্থায় আছে। কমপ্লিট হয়ে যাওয়া লাখ লাখ অর্ডারের জন্য ইনডেক্স তৈরি করে জায়গা নষ্ট করার কোনো মানে হয় না। এখানে আপনি শুধু status = 'pending' এর ওপর একটি Partial Index তৈরি করবেন।

### ২. এটি কেন এত শক্তিশালী?

সিস্টেম ডিজাইন করার সময় "Index Efficiency" ভাবতে হবে। Partial Index কেন ব্যবাহার করব তার ৩ টা প্রধান কারণ হচ্ছেঃ

**১. Reduced Storage:** ইনডেক্স ফাইলটি অনেক ছোট হয় কারণ এটি টেবিলের মাত্র ৫% বা ১০% ডেটা ধারণ করে।

**২. Faster Maintenance:** যখন আপনি টেবিলের কোনো রো আপডেট বা ইনসার্ট করেন যা আপনার ফিল্টার শর্তের বাইরে, তখন ডেটাবেজকে ইনডেক্স আপডেট করতে হয় না। এতে রাইট অপারেশন (Write Operation) ফাস্ট হয়।

**৩. Better Cache Performance:** ইনডেক্স ছোট হওয়ায় এটি র‍্যামে (RAM) থাকার সম্ভাবনা বেশি থাকে, ফলে সার্চিং হয় সুপারফাস্ট।

### ৩. Implementation (SQL Example)

ধরুন আপনার একটি users টেবিল আছে যেখানে লাখ লাখ ইউজার আছে, কিন্তু আপনি শুধু সেই ইউজারদের নিয়ে কাজ করেন যারা 'Active'।

**সাধারণ ইনডেক্স (বাজে প্র্যাকটিস যদি শুধু অ্যাক্টিভ ইউজার দরকার হয়):**

```sql
CREATE INDEX idx_all_users ON users(last_login);
-- এটি ইনঅ্যাক্টিভ ইউজারদের জন্যও ইনডেক্স তৈরি করবে, যা ডিস্ক স্পেস নষ্ট করবে।
```

**Partial Index (বেস্ট প্র্যাকটিস):**

```sql
CREATE INDEX idx_active_users ON users(last_login)
WHERE status = 'active';
-- এটি শুধু অ্যাক্টিভ ইউজারদের লিস্ট সর্টেড অবস্থায় রাখবে।
```

### ৪. সুবিধা ও সীমাবদ্ধতা

| সুবিধা (Pros)                                                   | সীমাবদ্ধতা (Cons)                                                                              |
| --------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| Storage Savings: বিশাল টেবিলের জন্য অনেক ডিস্ক স্পেস বাঁচায়।    | Specific Use Case: যদি আপনার কুয়েরি ফিল্টার শর্তের বাইরে যায়, তবে এই ইনডেক্স কোনো কাজ করবে না। |
| Write Performance: ইনসার্ট/আপডেট অনেক দ্রুত হয়।                 | Maintenance Complexity: ডেভেলপারকে মনে রাখতে হবে কোন শর্তে ইনডেক্সটি তৈরি করা হয়েছে।           |
| Less Fragmentation: ইনডেক্স ছোট হওয়ায় এটি দীর্ঘ সময় ক্লিন থাকে। | Query Matching: কুয়েরির WHERE ক্লজ এবং ইনডেক্সের WHERE ক্লজ হুবহু ম্যাচ করতে হয়।               |

### ৫. বাস্তব জীবনের সেরা উদাহরণ (Use Cases)

১. Soft Deletes: যদি আপনি deleted_at কলাম ব্যবহার করেন, তবে শুধু WHERE deleted_at IS NULL এর ওপর ইনডেক্স তৈরি করুন। ডিলিট হয়ে যাওয়া ডেটার ইনডেক্স রেখে লাভ নেই।

২. NULL Values: কোনো কলামে যদি প্রচুর NULL থাকে এবং আপনি শুধু ভ্যালু আছে এমন ডেটা খুঁজছেন (WHERE email IS NOT NULL)।

৩. Status Flags: যেমন is_processed = false, order_status = 'shipped' ইত্যাদি।

## নোটঃ

- **Unique Constraints:** আপনি Partial Index ব্যবহার করে চমৎকার ইউনিকনেস এনফোর্স করতে পারেন। যেমন: একজন ইউজারের শুধু একটিই 'Primary' অ্যাড্রেস থাকতে পারবে।

```sql
CREATE UNIQUE INDEX idx_single_primary_address
ON addresses (user_id) WHERE is_primary = true;
```

- **Optimizer Awareness:** মনে রাখবেন, আপনার কুয়েরিতে যদি ইনডেক্সের শর্তটি না থাকে, তবে ডেটাবেজ অপ্টিমাইজার এই ইনডেক্সটি ব্যবহার করবে না।
- **Don't Over-Filter:** শর্তটি খুব বেশি জটিল করবেন না। সহজ এবং ফ্রিকোয়েন্টলি ব্যবহৃত শর্ত ব্যবহার করুন।

## Summary

Partial Index হলো আপনার ডেটাবেজের জন্য একটি "সার্জিক্যাল স্ট্রাইক"। এটি বিশাল ডেটাসেটের ছোট একটি অংশকে টার্গেট করে সর্বোচ্চ পারফরম্যান্স নিশ্চিত করে।
