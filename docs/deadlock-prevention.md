## Deadlock Prevention: Designing Bulletproof Database Interactions

ডেডলক ডিটেকশন (Detection) হলো হার্ট অ্যাটাক হওয়ার পর চিকিৎসা করা, আর ডেডলক প্রিভেনশন (Prevention) হলো স্বাস্থ্যকর লাইফস্টাইল মেনে চলা যেন হার্ট অ্যাটাকই না হয়।

ডেডলক তখনই ঘটে যখন চারটি শর্ত (Mutual Exclusion, Hold and Wait, No Preemption, Circular Wait) একসাথে পূর্ণ হয়। Deadlock Prevention-এর মূল লক্ষ্য হলো এই শর্তগুলোর অন্তত একটিকে ভেঙে দেওয়া, যাতে ডেডলক হওয়ার সম্ভাবনা গাণিতিকভাবে শূন্য হয়ে যায়।

### ১. Overview (সহজ ভাষায়)

ডেডলক প্রিভেনশন মানে হলো আগে থেকেই এমন কিছু নিয়ম বা প্রোটোকল সেট করা যা ট্রানজেকশনগুলোকে একে অপরের পথে বাধা হয়ে দাঁড়াতে দেয় না।

- **Analogy:** একটি সরু রাস্তায় দুই দিক থেকে গাড়ি এসে মুখোমুখি দাঁড়িয়ে যাওয়া হলো ডেডলক। প্রিভেনশন হলো—রাস্তাটিকে 'ওয়ান-ওয়ে' করে দেওয়া অথবা ট্রাফিক সিগন্যাল বসানো যাতে এক দিকের গাড়ি যাওয়ার সময় অন্য দিক পুরোপুরি বন্ধ থাকে।

### ২. ডেডলক প্রিভেনশনের ৫টি গোল্ডেন রুল (The Senior's Playbook)

যখন হাই-ট্রাফিক সিস্টেম ডিজাইন করা হয় তখন এই মেথডগুলো ফলো করা হয়:

#### ১. Lock Order (সবচেয়ে কার্যকর উপায়)

সব ট্রানজেকশনকে সবসময় একই ক্রমানুসারে রিসোর্স লক করতে বাধ্য করা।

- **নিয়ম:** যদি আপনার Table A এবং Table B আপডেট করতে হয়, তবে সবসময় আগে A তারপর B আপডেট করুন।
- **কেন?** যদি সবাই একই সিরিয়াল মেনে চলে, তবে কখনো "Circular Wait" বা চক্রাকার অপেক্ষা তৈরি হবে না।

#### ২. Lock Hierarchy

বড় সিস্টেমে আমরা টেবিলগুলোকে লেভেলে ভাগ করি। একজন ইউজার কেবল তখনই উপরের লেভেলের লক নিতে পারবে যদি তার কাছে নিচের লেভেলের লক না থাকে। এটি অনেকটা লিফটের মতো—আপনি ৩ তলায় যাওয়ার আগে অবশ্যই ২ তলা পার হয়ে আসবেন।

#### ৩. Transaction Size (Keep it Small)

ট্রানজেকশন যত বড় হবে, সে তত বেশি রো লক করে রাখবে এবং ডেডলকের সম্ভাবনা তত বাড়বে।

- **টিপ:** বড় প্রসেসকে ছোট ছোট ট্রানজেকশনে ভাগ করুন। SELECT কোয়েরিকে ট্রানজেকশনের বাইরে রাখার চেষ্টা করুন যদি সেটি ক্রিটিক্যাল না হয়।

#### ৪. Acquire All Locks at Once

ট্রানজেকশন শুরু করার সময়ই যদি সব প্রয়োজনীয় লক একসাথে নিয়ে নেওয়া যায়, তবে "Hold and Wait" শর্তটি ভেঙে যায়। অর্থাৎ আপনার যদি ৩টি রো লাগে, তবে ৩টিই একবারে রিকোয়েস্ট করুন। যদি ১টি না পান, তবে অন্যগুলোও ধরবেন না।

#### ৫. No User Interaction inside Transactions

ট্রানজেকশন শুরু করার পর কখনো ইউজারের ইনপুট বা এক্সটার্নাল এপিআই (API) কলের জন্য অপেক্ষা করবেন না। ইউজার কফি খেতে গেলে আপনার ডেটাবেজ লক হয়ে বসে থাকবে, যা ডেডলকের পথ প্রশস্ত করে।

### ৩. ওয়ান-ওয়ে লজিক: Wait-Die vs Wound-Wait

আধুনিক ডিস্ট্রিবিউটেড সিস্টেমে ডেডলক এড়াতে ট্রানজেকশনের 'বয়স' (Timestamp) ব্যবহার করা হয়:

- **Wait-Die:** যদি একটি পুরনো (Older) ট্রানজেকশন নতুন (Younger) ট্রানজেকশনের লকের জন্য অপেক্ষা করে, তবে সে অপেক্ষা করতে পারে। কিন্তু যদি নতুন ট্রানজেকশন পুরনোটির জন্য অপেক্ষা করে, তবে নতুনটিকে 'মরে যেতে' (Abort) হয়।
- **Wound-Wait:** যদি পুরনো ট্রানজেকশন নতুনটির লক চায়, তবে সে নতুনটিকে 'আঘাত' (Kill) করে লক নিয়ে নেয়।

#### ৪. কোড লেভেল প্রিভেনশন (Example)

```javascript
// ❌ Bad: Order is random
async function transfer(from, to, amount) {
  await db.transaction(async (tx) => {
    await tx.updateAccount(from, -amount);
    await tx.updateAccount(to, amount);
  });
}

// ✅ Good: Always lock in ascending order of Account ID
async function transferOptimized(fromId, toId, amount) {
  const [firstId, secondId] = fromId < toId ? [fromId, toId] : [toId, fromId];

  await db.transaction(async (tx) => {
    await tx.lockAccount(firstId); // Always lock smaller ID first
    await tx.lockAccount(secondId);
    // ... perform updates
  });
}
```

### ৫. Pro-Tips

- **Optimistic Locking:** যেখানে সম্ভব সেখানে ডেটাবেজ লক না করে 'Version' কলাম ব্যবহার করুন। এটি ডেডলক হওয়ার সুযোগই দেয় না।
- **Avoid SELECT FOR UPDATE:** যদি খুব জরুরি না হয়, তবে এটি এড়িয়ে চলুন। এটি খুব শক্তিশালী লক তৈরি করে।
- **Set Lock Timeout:** ডাটাবেজ লেভেলে lock_timeout সেট করে রাখুন। এতে কোনো কুয়েরি অনন্তকাল আটকে না থেকে নির্দিষ্ট সময় পর এরর দিবে, যা আপনি কোড দিয়ে হ্যান্ডেল করতে পারবেন।

## Summary

- **Deadlock Prevention** হলো একটি প্রো-অ্যাক্টিভ অ্যাপ্রোচ।
- **সবচেয়ে সহজ এবং বেস্ট প্র্যাকটিস** হলো Consistent Locking Order।
- **ট্রানজেকশনকে সবসময় Atomic এবং Short রাখুন।**
- **সিস্টেম ডিজাইনের সময় Circular Dependency এড়িয়ে চলুন।**
