## Composite Index Order: The Art of Column Sequencing

> কম্পোজিট ইনডেক্স তৈরি করা আর একটি ডিকশনারি তৈরি করা একই কথা। আপনি যদি ডিকশনারিতে আগে 'অর্থ' দিয়ে তারপর 'শব্দ' সাজান, তবে কেউ কোনোদিন শব্দ খুঁজে পাবে না। ইনডেক্সে কলামের ক্রম বা Order ঠিক করাটাই হলো আসল ইঞ্জিনিয়ারিং।

একটি Composite Index (একাধিক কলামের ইনডেক্স) তখনই শক্তিশালী হয় যখন কলামগুলোর ক্রম সঠিকভাবে সাজানো থাকে। ভুল অর্ডারে ইনডেক্স তৈরি করলে ডেটাবেজ সেটি ব্যবহারই করতে পারবে না। একে বলা হয় "Left-to-Right" Rule।

### ১. Overview

কম্পোজিট ইনডেক্সকে একটি ফোনবুকের (Phonebook) সাথে তুলনা করুন। ফোনবুকে ডেটা সাজানো থাকে প্রথমে Last Name দিয়ে, তারপর First Name দিয়ে।

**সঠিক কুয়েরি:** আপনি যদি 'Reza, Nasim' খুঁজছেন, আপনি দ্রুত পাবেন। (Last Name জানা আছে)

**ভুল কুয়েরি:** আপনি যদি শুধু 'Nasim' খুঁজছেন (Last Name জানেন না), তবে আপনাকে পুরো ফোনবুক প্রথম থেকে শেষ পর্যন্ত খুঁজতে হবে।

ইনডেক্সের ক্ষেত্রেও ঠিক একই ঘটনা ঘটে। ডেটাবেজ বাম দিকের প্রথম কলামটি ছাড়া পরের কলামগুলোতে সরাসরি লাফ দিতে পারে না।

### ২. গোল্ডেন রুল: Left-to-Right Prefix

The database can only use a composite index if the leading (leftmost) column is in the WHERE clause.

ধরা যাক, আমাদের ইনডেক্সটি হলো: (Country, City, ZipCode)

| কুয়েরি (Query)                          | ইনডেক্স কি ব্যবহৃত হবে? | কেন?                                                         |
| --------------------------------------- | ----------------------- | ------------------------------------------------------------ |
| WHERE Country = 'BD'                    | Yes                     | প্রথম কলামটি আছে।                                            |
| WHERE Country = 'BD' AND City = 'Dhaka' | Yes                     | প্রথম দুটি কলাম ক্রমানুসারে আছে।                             |
| WHERE City = 'Dhaka'                    | No                      | প্রথম কলাম (Country) বাদ পড়েছে।                              |
| WHERE Country = 'BD' AND ZipCode = 1200 | Partial                 | শুধু Country অংশ কাজ করবে, ZipCode-এর জন্য স্ক্যান করতে হবে। |

### ৩. কীভাবে কলামের ক্রম (Order) ঠিক করবেন?

ইনডেক্স ডিজাইনের সময় এই ৩টি ফ্যাক্টর মাথায় রাখতে হবে:

**১. Selectivity (Cardinality)**
সবচেয়ে বেশি ইউনিক ভ্যালু যে কলামে আছে (High Cardinality), সেটিকে সাধারণত বামে রাখা হয়।

- **উদাহরণ:** (User_ID, Gender) ভালো, কিন্তু (Gender, User_ID) অতটা কার্যকর নয়, কারণ Gender মাত্র দুটি গ্রুপে ভাগ করে, যা সার্চ রেজাল্টকে খুব একটা ছোট করে না।

**২. Frequency of Use**
যে কলামটি আপনার অ্যাপ্লিকেশনের কুয়েরিতে সবচেয়ে বেশিবার WHERE ক্লজে ব্যবহৃত হয়, সেটিকে সবার বামে (First position) রাখুন।

**৩. Range Queries**
যদি আপনার কুয়েরিতে ইকুয়ালিটি (=) এবং রেঞ্জ (>, <, BETWEEN) উভয়ই থাকে, তবে ইকুয়ালিটি কলামগুলোকে বামে রাখুন এবং রেঞ্জ কলামকে ডানে।

- **সঠিক:** (Status, Created_At) — এখানে Status (Equality) আগে।

**৪. Implementation: SQL Example**

ধরুন আপনার একটি employees টেবিল আছে এবং আপনি প্রায়ই ডিপার্টমেন্ট এবং স্যালারি দিয়ে সার্চ করেন।

```sql
-- সিনিয়র লেভেল ইনডেক্স ডিজাইন
CREATE INDEX idx_dept_salary ON employees (department_id, salary);

-- এই কুয়েরিটি সুপারফাস্ট হবে
SELECT * FROM employees
WHERE department_id = 5 AND salary > 50000;

-- এই কুয়েরিতে ইনডেক্সটি কোনো কাজ করবে না (Index Skip)
SELECT * FROM employees
WHERE salary > 50000;
```

### ৫. সুবিধা ও সীমাবদ্ধতা

| সুবিধা (Pros)                                                                 | সীমাবদ্ধতা (Cons)                                                                 |
| ----------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| Multiple Filter Efficiency: একটি ইনডেক্স দিয়েই অনেকগুলো ফিল্টার কভার করা যায়। | Strict Ordering: অর্ডারে ভুল করলে ইনডেক্সটি অব্যবহৃত পড়ে থাকে।                    |
| Index-Only Scans: যদি সব কলাম ইনডেক্সে থাকে, তবে মেইন টেবিল রিড করতে হয় না।   | Redundancy: একাধিক কম্পোজিট ইনডেক্স তৈরি করলে ডিস্ক স্পেস এবং রাইট স্পিড কমে যায়। |

## নোটঃ

- **Order Matters in Equality too**: এমনকি যদি আপনি শুধু = ব্যবহার করেন, তবুও সবচেয়ে ফিল্টার্ড কলামটি বামে রাখা পারফরম্যান্স বাড়ায়।
- **Avoid Overlapping Indexes**: যদি আপনার (A, B) ইনডেক্স থাকে, তবে আলাদা করে শুধু (A) এর জন্য ইনডেক্স করার দরকার নেই। কারণ (A, B) ইনডেক্সটি (A) এর কাজও করে দিবে।
- **Check Explain Plan**: কুয়েরি শেষে EXPLAIN চালিয়ে দেখুন key_len কতটুকু ব্যবহৃত হচ্ছে। এটি আপনাকে বলে দিবে ইনডেক্সের কয়টি কলাম সার্চে অংশ নিয়েছে।

## Summary

কম্পোজিট ইনডেক্স হলো একটি চেইন। প্রথম লিঙ্ক (Leftmost column) ছিঁড়ে গেলে পুরো চেইনটি অকেজো হয়ে যায়। তাই ইনডেক্স তৈরির আগে আপনার কুয়েরি প্যাটার্ন ভালো করে স্টাডি করুন।
