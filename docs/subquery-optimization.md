## Subquery Optimization: Turning Slow Queries into Rockets

> সাবকুয়েরি (Subquery) লেখা খুব সহজ, কিন্তু সেটি অপ্টিমাইজ করা হলো আসল আর্ট। একটি অদক্ষ সাবকুয়েরি আপনার ডেটাবেজকে লুপের জালে আটকে ফেলে পারফরম্যান্স ধসিয়ে দিতে পারে।" অনেক সময় আমরা সাবকুয়েরি ব্যবহার করি শুধু কোড ছোট করার জন্য, কিন্তু পর্দার আড়ালে ডেটাবেজ ইঞ্জিনকে সেটি প্রসেস করতে অনেক ঘাম ঝরাতে হয়।

সাবকুয়েরি হলো একটি কুয়েরির ভেতরে আরেকটি কুয়েরি। এটি সাধারণত WHERE, FROM, বা SELECT ক্লজে ব্যবহৃত হয়। তবে সাবকুয়েরি ব্যবহারের আগে আমাদের বুঝতে হবে সেটি Correlated নাকি Non-correlated, কারণ এর ওপরই নির্ভর করে আপনার কুয়েরির স্পিড।

### ১. Subquery-র ধরণ এবং পারফরম্যান্স ইমপ্যাক্ট

### ক) Non-correlated Subquery (Independent)

এই সাবকুয়েরিটি মূল কুয়েরির ওপর নির্ভর করে না। এটি একবার রান হয় এবং রেজাল্টটি মেইন কুয়েরিকে দিয়ে দেয়।

**পারফরম্যান্স:** ভালো। কারণ এটি একবারই এক্সিকিউট হয়।

### খ) Correlated Subquery (Dependent)

এটি হলো পারফরম্যান্সের শত্রু। এই সাবকুয়েরিটি মেইন টেবিলের প্রতিটি রো-এর জন্য আলাদাভাবে রান হয়।

**পারফরম্যান্স:** খুব খারাপ। যদি মেইন টেবিলে ১ লাখ রো থাকে, তবে সাবকুয়েরিটি ১ লাখ বার রান হবে!

### ২. অপ্টিমাইজেশন টেকনিক: সাবকুয়েরি থেকে মুক্তি

**১. সাবকুয়েরি বনাম জয়েন (Join)**

অধিকাংশ ক্ষেত্রে একটি IN বা EXISTS সাবকুয়েরিকে JOIN দিয়ে রিপ্লেস করলে ডেটাবেজ অনেক ফাস্ট কাজ করে।

#### Bad (Correlated Subquery):

```sql
SELECT name FROM employees e
WHERE EXISTS (SELECT 1 FROM departments d WHERE d.id = e.dept_id AND d.name = 'IT');
```

#### Good (JOIN):

```sql
SELECT e.name FROM employees e
JOIN departments d ON e.dept_id = d.id
WHERE d.name = 'IT';
```

**কেন?** জয়েন ব্যবহার করলে ডেটাবেজ অপ্টিমাইজার (CBO) খুব সহজে 'Hash Join' বা 'Merge Join' অ্যালগরিদম ব্যবহার করতে পারে।

**২. IN বনাম EXISTS**

অনেকে মনে করেন দুটিই এক, কিন্তু এদের কাজ করার পদ্ধতি আলাদা:

- **IN:** সাধারণত সাবকুয়েরির পুরো রেজাল্ট সেট মেমোরিতে নিয়ে আসে। ছোট ডেটাসেটের জন্য ভালো।

- **EXISTS:** এটি শুধু চেক করে শর্তটি সত্য কি না। একটি ম্যাচ পাওয়ার সাথে সাথেই সে সার্চ বন্ধ করে দেয়। বড় ডেটাসেটের জন্য EXISTS সাধারণত দ্রুততর।

### ৩. Derived Tables বা CTE (Common Table Expressions)

যদি আপনার সাবকুয়েরিটি খুব জটিল হয়, তবে সেটিকে CTE বা WITH ক্লজ দিয়ে আলাদা করে ফেলুন। এতে কোড যেমন পরিষ্কার হয়, ডেটাবেজও মাঝেমধ্যে এটিকে মেটেরিয়ালাইজ (Materialize) করে দ্রুত রান করতে পারে।

**Professional Style:**

```sql
WITH HighEarners AS (
    SELECT dept_id, AVG(salary) as avg_sal
    FROM employees
    GROUP BY dept_id
)
SELECT e.name, h.avg_sal
FROM employees e
JOIN HighEarners h ON e.dept_id = h.dept_id
WHERE e.salary > h.avg_sal;
```

### ৪. Performance Tips

১. Avoid Subqueries in SELECT: SELECT ক্লজের ভেতরে সাবকুয়েরি লেখা এড়িয়ে চলুন। এটি প্রতিটি রো-এর জন্য একবার করে রান হয়, যা আপনার কুয়েরিকে পঙ্গু করে দিতে পারে। এর বদলে LEFT JOIN ব্যবহার করুন।

২. Check the Explain Plan: কুয়েরি স্লো হলে EXPLAIN কমান্ড দেখুন। যদি সাবকুয়েরির জায়গায় "Dependent Subquery" লেখা দেখেন, তবে বুঝবেন আপনার কুয়েরি অপ্টিমাইজ করা জরুরি।

৩. Unnesting: আধুনিক ডেটাবেজ অপ্টিমাইজার নিজে থেকেই অনেক সাবকুয়েরিকে জয়েনে রূপান্তর করে নেয় (একে বলে 'Query Flattening')। তবে সবসময় ইঞ্জিনের ওপর ভরসা না করে কোড লেভেলেই অপ্টিমাইজ করা নিরাপদ।

### ৫. কখন সাবকুয়েরি ব্যবহার করবেন?

| পরিস্থিতি              | সাবকুয়েরি কি ব্যবহারযোগ্য? | বিকল্প                                  |
| ---------------------- | -------------------------- | --------------------------------------- |
| ছোট রেজাল্ট সেট        | Yes                        | সরাসরি সাবকুয়েরি সহজ।                   |
| বড় টেবিল ফিল্টারিং     | ❌ No                      | JOIN অথবা EXISTS ব্যবহার করুন।          |
| অ্যানালিটিক্যাল কুয়েরি | Careful                    | Window Functions (OVER()) ব্যবহার করুন। |

## নোটঃ

- **Think Set-based, not Row-based:** ডেটাবেজ লুপের মতো চিন্তা করে না, সে সেট (Set) নিয়ে কাজ করে। সাবকুয়েরি অনেক সময় রো-বেজড চিন্তার ফসল, যা অপ্টিমাইজ করা প্রয়োজন।
- **Index the Keys:** সাবকুয়েরি বা জয়েন যাই ব্যবহার করুন না কেন, যদি ফরেন কি (Foreign Key)-তে ইনডেক্স না থাকে, তবে কোনো অপ্টিমাইজেশনেই কাজ হবে না।
- **Limit the Columns:** সাবকুয়েরির ভেতরে SELECT \* ব্যবহার করবেন না। শুধু সেই কলামটি নিন যা আপনার দরকার।

## Summary

সাবকুয়েরি হলো একটি শক্তিশালী টুল, কিন্তু ভুল ব্যবহারে এটি পারফরম্যান্স কিলার। "যদি সাবকুয়েরিটি জয়েন দিয়ে লেখা সম্ভব হয়, তবে জয়েনই ব্যবহার করুন।" এটি আপনার ডেটাবেজ অপ্টিমাইজারকে সেরা প্ল্যানটি তৈরি করতে সাহায্য করবে।
