## Surrogate Key vs Natural Key: Choosing the Right Identity

> ডেটাবেজ টেবিলের প্রতিটি রেকর্ডকে ইউনিকভাবে চেনার জন্য আমরা প্রাইমারি কি (Primary Key) ব্যবহার করি। এই কি বা চাবি দুই ধরণের হতে পারে— Surrogate এবং Natural। একজন প্রোগ্রামার হিসেবে আপনাকে জানতে হবে কখন কোনটি ব্যবহার করা নিরাপদ।

### ১. Natural Key (প্রাকৃতিক চাবি)

**Natural Key** হলো এমন একটি ডেটা যা বাস্তব জগত (Real World) থেকে আসে এবং যা স্বভাবগতভাবেই ইউনিক।

- **উদাহরণ:** NID Number, Passport Number, বা Email Address (তাত্ত্বিকভাবে)।
- **বৈশিষ্ট্য:** এটি বিজনেস লজিকের অংশ। এটি শুধু ডেটাবেজের জন্য তৈরি করা হয়নি, বরং বাইরের জগতেও এর অস্তিত্ব আছে।

### ২. Surrogate Key (সারোগেট বা কৃত্রিম চাবি)

**Surrogate Key** হলো একটি কৃত্রিম আইডি যা সিস্টেম নিজে জেনারেট করে। এর সাথে বাস্তব জগতের কোনো সম্পর্ক নেই।

- **উদাহরণ:** Auto-incremented ID (1, 2, 3...) বা UUID/GUID।
- **বৈশিষ্ট্য:** এটি সম্পূর্ণ অর্থহীন (Meaningless) ডেটা। এর একমাত্র কাজ হলো রেকর্ডটিকে ইউনিকলি আইডেন্টিফাই করা।

### ৩. The Great Debate: পার্থক্য ও তুলনা

| বৈশিষ্ট্য    | Natural Key                                               | Surrogate Key                             |
| ------------ | --------------------------------------------------------- | ----------------------------------------- |
| উৎস          | বাস্তব জগত থেকে আসে।                                      | সিস্টেম জেনারেট করে।                      |
| বিজনেস লজিক  | বিজনেস রুলস পরিবর্তন হলে এটি বদলে যেতে পারে।              | এটি কখনোই বদলায় না (Immutable)।           |
| পারফরম্যান্স | স্ট্রিং বা বড় নম্বর হলে ইনডেক্সিং এবং জয়েন স্লো হতে পারে। | ইন্টিজার (Integer) হলে জয়েন খুব ফাস্ট হয়। |
| স্টোরেজ      | বেশি জায়গা নিতে পারে (যেমন- Passport No)।                 | খুব কম জায়গা নেয় (৪ বা ৮ বাইট)।           |

#### বেশিরভাগ আধুনিক আর্কিটেকচারে আমরা Surrogate Key (যেমন- id) পছন্দ করি। কেন?

**১. স্থায়িত্ব (Stability):** ধরুন আপনি Email কে ন্যাচারাল কি বানালেন। এখন ইউজার যদি তার ইমেইল পরিবর্তন করতে চায়, তবে আপনাকে ওই ইউজারের সাথে যুক্ত সব টেবিলের ফরেন কি (Foreign Key) আপডেট করতে হবে। এটি একটি বিশাল পেইন! কিন্তু সারোগেট কি থাকলে ইউজার তার ইমেইল বা ফোন নম্বর হাজারবার বদলালেও আপনার ডেটাবেজ রিলেশনশিপে কোনো প্রভাব পড়বে না।

**২. সিকিউরিটি:** ন্যাচারাল কি অনেক সময় সেনসিটিভ ডেটা (যেমন- NID) প্রকাশ করে দেয়। সারোগেট কি কোনো ইনফরমেশন লিক করে না।

### ৪. SQL Implementation

Natural Key Example:

```sql
    CREATE TABLE Users (
    passport_no VARCHAR(20) PRIMARY KEY, -- Natural Key
    full_name VARCHAR(100)
);
```

Surrogate Key Example:

```sql
    CREATE TABLE Users (
    id INT AUTO_INCREMENT PRIMARY KEY, -- Surrogate Key
    passport_no VARCHAR(20) UNIQUE,     -- একে শুধু ইউনিক রাখা ভালো
    full_name VARCHAR(100)
);
```

### ৫. Best Practices

- **Rule of Thumb:** সবসময় একটি id (Surrogate Key) কলাম রাখুন। আর ন্যাচারাল কি-কে (যেমন- Email বা NID) UNIQUE কনস্ট্রেইন্ট হিসেবে সেট করুন।
- **UUID vs Integer:** যদি আপনার সিস্টেম ডিস্ট্রিবিউটেড হয় (অনেকগুলো সার্ভার), তবে UUID ব্যবহার করুন। ছোট অ্যাপ্লিকেশনের জন্য BIGINT AUTO_INCREMENT যথেষ্ট।
- **Don't attach meaning:** প্রাইমারি কি-র মধ্যে কোনো অর্থ খুঁজবেন না। এটি শুধু একটি ট্যাগ বা লেবেল।
