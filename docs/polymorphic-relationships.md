## Polymorphic Relationships: The Universal Connector

> ডেটাবেজ ডিজাইনে আমরা সাধারণত 1:1, 1:N, বা M:N রিলেশনশিপ দেখি। কিন্তু এমন কিছু পরিস্থিতি আসে যেখানে একটি টেবিলকে একাধিক আলাদা আলাদা টেবিলের সাথে যুক্ত হতে হয়। সেখানেই আমরা Polymorphic Relationship ব্যবহার করি।

### ১. Overview (সহজ ভাষায়)

**Polymorphic Relationship** হলো এমন একটি সিস্টেম যেখানে একটি চাইল্ড টেবিল (Child Table) একক সম্পর্কের মাধ্যমে একাধিক প্যারেন্ট টেবিলের (Parent Table) সাথে যুক্ত হতে পারে।

- **সাধারণ রিলেশন:** একটি `Comment` শুধু একটি `Post`-এর হতে পারে।
- **পলিমরফিক রিলেশন:** একটি `Comment` একই সাথে একটি `Post`, একটি `Video`, অথবা একটি `Product`-এরও হতে পারে।

### ২. সমস্যাটি কোথায়? (The Problem)

ধরুন, আপনার সিস্টেমে ইউজাররা `Post` এবং `Video`—উভয় জায়গাতেই কমেন্ট করতে পারে। পলিমরফিক রিলেশন ছাড়া আপনাকে নিচের মতো ডিজাইন করতে হতো:

| id  | body        | post_id | video_id |
| --- | ----------- | ------- | -------- |
| 1   | Great post! | 101     | NULL     |
| 2   | Nice video! | NULL    | 505      |

**সমস্যা:** এখানে অনেকগুলো কলাম NULL হয়ে পড়ে থাকে। ভবিষ্যতে যদি আপনি Product বা Image টেবিলেও কমেন্ট যোগ করতে চান, তবে আপনাকে বারবার টেবিল অল্টার (Alter) করে নতুন কলাম যোগ করতে হবে। এটি মোটেও স্কেলেবল নয়।

### ৩. সমাধান: পলিমরফিক ডিজাইন (The Solution)

পলিমরফিক রিলেশনে আমরা দুটি স্পেশাল কলাম ব্যবহার করি:

**১. able_id:** প্যারেন্ট রেকর্ডের আইডি স্টোর করে।

**২. able_type:** প্যারেন্ট টেবিলের নাম বা মডেলের নাম স্টোর করে।

```sql
CREATE TABLE comments (
    id INT AUTO_INCREMENT PRIMARY KEY,
    body TEXT,
    commentable_id INT,        -- এটি হতে পারে Post ID বা Video ID
    commentable_type VARCHAR(50) -- এখানে থাকবে 'Post' অথবা 'Video'
);
```

### ৪. At a Glance (সুবিধা অসুবিধা)

| সুবিধা (Pros)                                                                             | সীমাবদ্ধতা (Cons)                                                                                            |
| ----------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| Flexibility: ভবিষ্যতে নতুন কোনো টেবিলের সাথে কমেন্ট যুক্ত করতে টেবিল পরিবর্তন করতে হয় না। | No Foreign Keys: সরাসরি ডেটাবেজ লেভেলে Foreign Key এনফোর্স করা যায় না, কারণ আইডিগুলো বিভিন্ন টেবিল থেকে আসে। |
| Cleaner Schema: অনেকগুলো NULL কলামের প্রয়োজন হয় না।                                       | Query Complexity: কুয়েরি করার সময় লজিক কিছুটা জটিল হতে পারে।                                                 |
| DRY Principle: একই লজিক বারবার লিখতে হয় না।                                               | Data Integrity: প্যারেন্ট রেকর্ড ডিলিট হলে চাইল্ড রেকর্ডগুলো ম্যানুয়ালি ক্লিন করতে হয়।                       |

### ৫. বাস্তব জীবনের উদাহরণ (Use Cases)

**১. Comments:** পোস্ট, ভিডিও বা ইভেন্টে কমেন্ট করা।

**২. Images/Media:** ইউজার প্রোফাইল পিকচার, প্রোডাক্ট ইমেজ বা পোস্ট ইমেজ—সবই একটি images টেবিলে রাখা।

**৩. Tags:** ব্লগ পোস্ট এবং প্রোডাক্ট—উভয়কেই ট্যাগ করা।

**৪. Likes/Reactions:** বিভিন্ন ধরণের কন্টেন্ট লাইক করা।

## নোটঃ

- **Naming Convention:** সাধারণত লারাভেল (Laravel) বা অন্য ফ্রেমেওয়ার্কে টেবিলের নামের শেষে able যুক্ত করা হয় (যেমন: commentable, taggable, likeable)। এটি রিলেশনটি চিনতে সাহায্য করে।

- **Indexing:** able_id এবং able_type কলাম দুটির ওপর Composite Index ব্যবহার করুন। এতে সার্চ পারফরম্যান্স অনেক বেড়ে যাবে।

- **Application Logic:** যেহেতু ডেটাবেজ লেভেলে ফরেন কি নেই, তাই অ্যাপ্লিকেশনের কোড লেভেলে ডেটা ক্লিনিং (যেমন: onDelete লজিক) নিশ্চিত করুন।
