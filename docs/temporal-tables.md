## Temporal Tables: The Database Time Machine

> ডেটাবেজের সাধারণ টেবিলে আমরা সাধারণত শুধু বর্তমান (Current State) ডেটা দেখতে পাই। কিন্তু যদি আপনার এমন একটি সিস্টেম প্রয়োজন হয় যেখানে আপনি চাইলেই অতীত বা কোনো নির্দিষ্ট সময়ের ডেটা দেখতে পারবেন, তবে আপনাকে Temporal Tables বা System-Versioned Tables ব্যবহার করতে হবে।

### ১. Overview (সহজ ভাষায়)

Temporal Tables হলো এমন এক বিশেষ ধরণের টেবিল যা অটোমেটিকভাবে ডেটার পরিবর্তনের ইতিহাস (History) জমা রাখে। এটি অনেকটা আপনার ফাইল সিস্টেমের "Undo" বা "Git History"-র মতো কাজ করে।

- **সাধারণ টেবিল**: নতুন ভ্যালু আপডেট করলে পুরনোটা হারিয়ে যায়।
- **টেম্পোরাল টেবিল**: পুরনো ভ্যালুটি একটি ব্যাকআপ (History) টেবিলে চলে যায় এবং বর্তমান টেবিলে নতুন ভ্যালু আপডেট হয়।

### ২. এটি কীভাবে কাজ করে? (The Mechanism)

- **Period Columns**: প্রতিটি রো-তে দুটি কলাম থাকে— `SysStartTime` এবং `SysEndTime` যা নির্দেশ করে ওই ডেটাটি কখন থেকে কখন পর্যন্ত বৈধ ছিল।
- **History Table**: একটি শ্যাডো টেবিল যা মূল টেবিলের সাথে হুবহু এক, যেখানে পুরনো সব ডেটা স্টোর হয়।

যখন আপনি কোনো রো Update করেন:

- বর্তমান রো-টির `SysEndTime` আপডেট হয়ে 'এখনকার সময়' হয়ে যায় এবং সেটি হিস্টোরি টেবিলে চলে যায়।
- নতুন ডেটাটি বর্তমান টেবিলে ইনসার্ট হয় যার `SysStartTime` শুরু হয় 'এখন' থেকে।

### ৩. Architectural Implementation (SQL Example)

```sql
CREATE TABLE Products (
    ProductID INT PRIMARY KEY,
    ProductName NVARCHAR(100),
    Price DECIMAL(10, 2),
    -- পিরিয়ড কলাম ডিফাইন করা
    SysStartTime DATETIME2 GENERATED ALWAYS AS ROW START,
    SysEndTime DATETIME2 GENERATED ALWAYS AS ROW END,
    PERIOD FOR SYSTEM_TIME (SysStartTime, SysEndTime)
)
WITH (SYSTEM_VERSIONING = ON); -- হিস্টোরি ম্যানেজমেন্ট অন করা
```

**Time Travel Query:**
আপনি যদি দেখতে চান গত ১লা জানুয়ারি এই প্রোডাক্টের দাম কত ছিল:

```sql
SELECT * FROM Products
FOR SYSTEM_TIME AS OF '2026-01-01 12:00:00';
```

### ৪. At a Glance (সুবিধা অসুবিধা)

| সুবিধা (Pros)                                                             | সীমাবদ্ধতা (Cons)                                                                 |
| ------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| Audit Log: কে কখন ডেটা চেঞ্জ করেছে তার পারফেক্ট ট্র্যাক থাকে।             | Storage Bloat: হিস্টোরি টেবিল খুব দ্রুত অনেক বড় হয়ে যেতে পারে।                    |
| Data Recovery: ভুল করে ডেটা ডিলিট বা আপডেট করলে সহজেই ফিরে আসা যায়।       | Performance Hit: প্রতিটি রাইট অপারেশনে ডেটা কপি করতে হয় বলে কিছুটা স্লো হতে পারে। |
| Compliance: ফিন্যান্সিয়াল বা মেডিকেল ডেটার ক্ষেত্রে এটি আইনি বাধ্যবাধকতা। | Schema Changes: মূল টেবিলের স্ট্রাকচার চেঞ্জ করা কিছুটা জটিল হয়ে যায়।             |

### ৫. বাস্তব জীবনের উদাহরণ (Use Cases)

**১. Banking System:** ব্যাংক অ্যাকাউন্টের ব্যালেন্স কোনো নির্দিষ্ট দিনে কত ছিল তা জানতে।

**২. Subscription Platform:** একজন ইউজার আগে কোন প্যাকেজটি ব্যবহার করতেন তা ট্র্যাক করতে।

**৩. Inventory Management:** স্টক কতবার পরিবর্তন হয়েছে তার রিপোর্ট তৈরি করতে।

## নোটঃ

- **Purge Strategy:** হিস্টোরি টেবিল যেন ইনফিনিটি পর্যন্ত না বাড়ে, তাই পুরনো হিস্টোরি ডেটা আর্কাইভ বা ডিলিট করার জন্য একটি 'Retention Policy' রাখুন।

- **Indexing:** হিস্টোরি টেবিলের SysStartTime এবং SysEndTime এর ওপর ইনডেক্সিং করতে ভুলবেন না, নতুবা টাইম-ট্রাভেল কুয়েরি স্লো হয়ে যাবে।

- **Naming Convention:** হিস্টোরি টেবিলগুলোর নাম সাধারণত \_History সাফিক্স দিয়ে রাখা ভালো।
