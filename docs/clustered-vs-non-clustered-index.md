## Clustered vs Non-clustered Index: The Physical Storage Strategy

ইনডেক্সিং শুধু ডেটা খোঁজা নয়, এটি হলো হার্ড ড্রাইভের ভেতরে ডেটাগুলো আসলে কীভাবে সাজানো থাকবে তার ফিজিক্যাল লেআউট। আপনি যদি Clustered এবং Non-clustered ইনডেক্সের পার্থক্য না বোঝেন, তবে আপনার কোড হয়তো কাজ করবে, কিন্তু স্কেলেবল হবে না।

ডেটাবেজ যখন ডিস্কে ডেটা সেভ করে, তখন সেটিকে সাজানোর দুটি প্রধান উপায় আছে। একটি হলো ডেটাগুলোকে Physically লাইনে দাঁড় করানো, আর অন্যটি হলো একটি আলাদা ম্যাপ বা অ্যাড্রেস বুক তৈরি করা।

### ১. Clustered Index (সরাসরি ডেটা)

Clustered Index নির্ধারণ করে একটি টেবিলের ডেটাগুলো ডিস্কের ভেতরে ফিজিক্যালি কীভাবে সাজানো থাকবে।

- **সহজ উদাহরণ:** একটি ডিকশনারি বা অভিধান। এখানে শব্দগুলো বর্ণানুক্রমিক (A-Z) সাজানো থাকে। আপনি যদি 'Apple' খুঁজছেন, তবে আপনি জানেন সেটি 'A' সেকশনে আছে এবং ওই শব্দটির পাশেই তার অর্থ (Data) লেখা আছে।
- **বৈশিষ্ট্য:** একটি টেবিলে মাত্র একটি Clustered Index থাকতে পারে। কারণ ডেটাগুলো ডিস্কে শুধু একভাবেই সাজানো সম্ভব।
- **ডিফল্ট:** সাধারণত কোনো কলামকে Primary Key বানালে ডেটাবেজ সেটিকে অটোমেটিক Clustered Index হিসেবে সেট করে।

### ২. Non-clustered Index (অ্যাড্রেস বুক)

Non-clustered Index হলো মূল ডেটা থেকে আলাদা একটি স্ট্রাকচার। এটি ডেটার ফিজিক্যাল অর্ডার পরিবর্তন করে না, বরং একটি পয়েন্টার (Pointer) বা অ্যাড্রেস জমা রাখে।

- **সহজ উদাহরণ:** একটি টেক্সট বইয়ের শেষের দিকের "Index" পেজ। সেখানে টপিকগুলো বর্ণানুক্রমিক সাজানো থাকে এবং পাশে একটি পেজ নম্বর থাকে। আপনি ইনডেক্স থেকে টপিক দেখে পেজ নম্বরে (Pointer) জাম্প করেন।
- **বৈশিষ্ট্য:** একটি টেবিলে একাধিক (সাধারণত ২৫০+ পর্যন্ত) Non-clustered Index থাকতে পারে।
- **সঞ্চয়স্থান:** এটি মূল টেবিল থেকে আলাদা জায়গায় স্টোর হয়, তাই এটি ডিস্কে অতিরিক্ত জায়গা নেয়।

### ৩. Key Differences: এক নজরে পার্থক্য

| বৈশিষ্ট্য      | Clustered Index                          | Non-clustered Index                                                |
| -------------- | ---------------------------------------- | ------------------------------------------------------------------ |
| Physical Order | ডেটাগুলো এই ইনডেক্স অনুযায়ী সাজানো থাকে। | ডেটার ফিজিক্যাল অর্ডারের সাথে সম্পর্ক নেই।                         |
| Leaf Node      | লিফ নোডেই একচুয়াল ডেটা থাকে।             | লিফ নোডে শুধু ডেটার পয়েন্টার বা অ্যাড্রেস থাকে।                    |
| সংখ্যা         | একটি টেবিলে মাত্র ১টি থাকতে পারে।        | একটি টেবিলে অনেকগুলো থাকতে পারে।                                   |
| Search Speed   | অত্যন্ত দ্রুত (সরাসরি ডেটা পাওয়া যায়)।   | কিছুটা ধীর (আগে ইনডেক্স থেকে অ্যাড্রেস নিয়ে তারপর ডেটাতে যেতে হয়)। |
| Storage        | আলাদা কোনো এক্সট্রা স্পেস লাগে না।       | আলাদা ডিরেক্টরি হিসেবে এক্সট্রা স্পেস লাগে।                        |

### ৪. Performance Tip: The "Bookmark" Logic

যখন আপনি Non-clustered Index ব্যবহার করে কিছু খোঁজেন, তখন ডেটাবেজ প্রথমে ইনডেক্স ফাইলটি পড়ে, সেখান থেকে 'Row ID' বা 'Clustered Key' সংগ্রহ করে, তারপর আবার মেইন টেবিল থেকে ডেটা তুলে আনে। একে বলা হয় Key Lookup। যদি আপনার কুয়েরিতে অনেক বেশি ডেটা রিড করতে হয়, তবে Clustered Index সবসময় জয়ী হবে কারণ সেখানে কোনো লুকআপ ওভারহেড নেই।

### ৫. বাস্তব জীবনের উদাহরণ (Use Cases)

১. Primary Key (ID): এটি সবসময় Clustered Index হওয়া উচিত যাতে আইডি দিয়ে সার্চ করলে দ্রুততম রেজাল্ট আসে।

২. Range Queries (Date/Salary): যেখানে BETWEEN কুয়েরি বেশি হয়, সেখানে Clustered Index দারুণ কারণ ডেটাগুলো ডিস্কে পাশাপাশি সাজানো থাকে।

৩. Foreign Keys / Filters (Email/Category): এই কলামগুলোতে Non-clustered Index দেওয়া হয় কারণ আমরা ইমেইল দিয়ে সার্চ করতে চাই, কিন্তু ডেটাগুলো ইমেইল অনুযায়ী সাজানোর দরকার নেই।

## নোটঃ

- **Don't Over-Index**: অনেক বেশি Non-clustered Index দিলে আপনার INSERT এবং UPDATE অপারেশন অনেক স্লো হয়ে যাবে, কারণ প্রতিটি পরিবর্তনের পর ডেটাবেজকে সব ইনডেক্স ফাইল আপডেট করতে হয়।
- **Index Fragmentation**: অনেক বেশি ডেটা ডিলিট বা ইনসার্ট করলে Clustered Index অগোছালো হয়ে যেতে পারে। মাঝে মাঝে ইনডেক্স Rebuild করা পারফরম্যান্সের জন্য ভালো।
- **Choose Composite Keys Wisely**: যদি আপনি একাধিক কলাম মিলে ইনডেক্স করেন, তবে সবচেয়ে বেশি ব্যবহৃত কলামটি বামে রাখুন।

## Summary

Clustered Index হলো বই নিজেই, যেখানে কন্টেন্ট সাজানো আছে। আর Non-clustered Index হলো বইয়ের পেছনের ইনডেক্স পেজ, যা আপনাকে কন্টেন্টের ঠিকানায় নিয়ে যায়।
