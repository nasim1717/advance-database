## Multi-tenant Data Modeling: Building for the SaaS World

> আমরা যখন SaaS অ্যাপ্লিকেশন বানানোর চিন্তা করি, তখন আমাদের লক্ষ্য থাকে একটি মাত্র কোডবেস এবং ইনফ্রাস্ট্রাকচার ব্যবহার করে অসংখ্য আলাদা গ্রাহককে সার্ভিস দেওয়া। কিন্তু চ্যালেঞ্জটা হলো— ডেটা আইসোলেশন (Data Isolation)। অর্থাৎ, এক ট্যানেন্টের ডেটা যেন অন্য ট্যানেন্ট কোনোভাবেই দেখতে না পারে।

### ১. Overview (সহজ ভাষায়)

মাল্টি-ট্যানেন্সি অনেকটা একটি অ্যাপার্টমেন্ট বিল্ডিংয়ের মতো।

- বিল্ডিংয়ের ইনফ্রাস্ট্রাকচার (লিফট, পানির লাইন) সবাই শেয়ার করে।
- কিন্তু প্রতিটি পরিবারের জন্য আলাদা ফ্ল্যাট আছে, যেখানে তাদের গোপনীয়তা বজায় থাকে।

সফটওয়্যারের ক্ষেত্রে এটি ৩টি প্রধান উপায়ে ডিজাইন করা যায়।

### ২. ৩টি প্রধান আর্কিটেকচার (The 3 Strategies)

#### ক) Shared Database, Shared Schema (সবার জন্য এক টেবিল)

সব ট্যানেন্টের ডেটা একই টেবিলে থাকে। প্রতিটি টেবিলে একটি tenant_id কলাম থাকে যা দিয়ে ডেটা আলাদা করা হয়।

**Note:** এটি সবচেয়ে সস্তা এবং সহজ পদ্ধতি। কিন্তু ডেভেলপার যদি কোনো কুয়েরিতে WHERE tenant_id = ? দিতে ভুলে যায়, তবে ডেটা লিক হওয়ার বিশাল ঝুঁকি থাকে।

#### খ) Shared Database, Separate Schema (আলাদা রুম, এক বাড়ি)

ডেটাবেজ একটাই থাকে, কিন্তু প্রতিটি ট্যানেন্টের জন্য আলাদা Schema (যেমন PostgreSQL-এ) ব্যবহার করা হয়।

**Note:** এটি একটি ব্যালেন্সড পদ্ধতি। এতে সিকিউরিটি ভালো থাকে এবং কোড লেভেলে কুয়েরি করা সহজ হয়।

#### গ) Separate Database (পুরো আলাদা বাড়ি)

প্রতিটি ট্যানেন্টের জন্য আলাদা ফিজিক্যাল ডেটাবেজ সার্ভার বা ডেটাবেজ থাকে।

**Note:** এটি সবচেয়ে দামী কিন্তু সবচেয়ে নিরাপদ। ব্যাংকিং বা অত্যন্ত সেনসিটিভ ডেটার ক্ষেত্রে এটি ব্যবহার করা হয়।

### ৩. Implementation Example (Shared Schema Approach)

জুনিয়র হিসেবে আমরা হয়ত প্রথম পদ্ধতিটি ব্যবহার করব বা করি। নিচে তার একটি উদাহরণ:

```sql
CREATE TABLE orders (
    id INT AUTO_INCREMENT PRIMARY KEY,
    tenant_id INT NOT NULL, -- এটিই সেই 'Universal Adapter'
    product_name VARCHAR(100),
    amount DECIMAL(10, 2),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX (tenant_id) -- এই ইনডেক্সটি মাস্ট!
);
```

কুয়েরি করার সময়:

```sql
SELECT * FROM orders WHERE tenant_id = CURRENT_TENANT_ID();
```

### ৪. তুলনামূলক আলোচনা (Comparison)

| বৈশিষ্ট্য       | Shared Schema              | Separate Schema         | Separate DB             |
| --------------- | -------------------------- | ----------------------- | ----------------------- |
| খরচ (Cost)      | খুব কম (Low)               | মাঝারি (Medium)         | অনেক বেশি (High)        |
| সিকিউরিটি       | রিস্কি (Low Isolation)     | ভালো (Medium Isolation) | সেরা (Strong Isolation) |
| স্কেলেবিলিটি    | অনেক ট্যানেন্ট রাখা যায়    | ম্যানেজ করা কিছুটা কঠিন | মেইনটেইন করা খুব কঠিন   |
| ব্যাকআপ/রিস্টোর | একজনের ডেটা আলাদা করা কঠিন | সহজ                     | সবচেয়ে সহজ              |

### ৫. বাস্তব জীবনের উদাহরণ (Use Cases)

- **Slack**: প্রতিটি কোম্পানির জন্য আলাদা ওয়ার্কস্পেস (Shared Schema বা Separate Schema)।
- **Shopify**: প্রতিটি দোকানের জন্য আলাদা সাব-ডোমেইন এবং ডেটাবেজ ম্যানেজমেন্ট।
- **Enterprise Software**: বড় ক্লায়েন্টের জন্য যারা তাদের ডেটা অন্য কারো সাথে একই সার্ভারে রাখতে চায় না (Separate DB)।

## নোটঃ

- Row-Level Security (RLS): আপনি যদি PostgreSQL ব্যবহার করেন, তবে ডেটাবেজ লেভলেই RLS এনাবল করতে পারেন। এতে অ্যাপ্লিকেশনের কোডে ভুল করলেও ডেটাবেজ অটোমেটিক tenant_id ফিল্টার করে দিবে।
- Global Scope: লারাভেলের (Laravel) মতো ফ্রেমওয়ার্কে Global Scopes ব্যবহার করুন যাতে প্রতিবার ম্যানুয়ালি WHERE tenant_id লিখতে না হয়।
- Tenant Migration: আলাদা স্কিমা বা ডেটাবেজ ব্যবহার করলে মাইগ্রেশন চালানো (যেমন- নতুন টেবিল যোগ করা) চ্যালেঞ্জিং হতে পারে। এজন্য অটোমেশন স্ক্রিপ্ট তৈরি করে রাখুন।
