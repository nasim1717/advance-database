## Composite Primary Key: The Power of Multiple Columns

> ডেটাবেজ ডিজাইনে আমরা সাধারণত অভ্যস্ত যে প্রতিটি টেবিলে একটি `id` কলাম থাকবে। কিন্তু বাস্তব জগতের জটিল ডেটা মডেলিংয়ে সবসময় একটি সিঙ্গেল আইডি যথেষ্ট নয়। সেখানেই জন্ম হয় **Composite Primary Key**-র।

### ১. Overview (সংক্ষিপ্ত পরিচিতি)

যখন দুই বা ততোধিক কলাম একত্রে মিলে একটি টেবিলের প্রতিটি রো (Row) বা রেকর্ডকে ইউনিকলি (Uniquely) আইডেন্টিফাই করে, তখন তাকে **Composite Primary Key** বলা হয়।

- **সিম্পল পিকে (Simple PK):** শুধু একটি কলাম (যেমন: `student_id`)।
- **কম্পোজিট পিকে (Composite PK):** দুই বা তার বেশি কলামের কম্বিনেশন (যেমন: `student_id + semester_id`)।

### ২. কেন এটি প্রয়োজন? (The "Why")

আমরা কম্পোজিট কি তখনই ব্যবহার করি যখন ডেটার ইউনিকনেস তার সম্পর্কের (Relationship) ওপর নির্ভর করে।

**উদাহরণ:** একটি ক্লাসরুমের কথা চিন্তা করুন।

- আপনার রোল নম্বর ১০ হতে পারে।
- পাশের ক্লাসেও কারো রোল নম্বর ১০ হতে পারে।
- কিন্তু "ক্লাস" এবং "রোল" এই দুটি তথ্য যখন একসাথে করা হয় (Class 10, Roll 10), তখন সেটি সারা স্কুলে ইউনিক হয়ে যায়।

### ৩. Architectural Implementation (SQL Example)

```sql
CREATE TABLE Course_Registration (
    student_id INT,
    course_id INT,
    registration_date DATE,
    -- এখানে student_id এবং course_id মিলে Composite PK তৈরি করছে
    PRIMARY KEY (student_id, course_id)
);
```

এখানে student_id বা course_id আলাদাভাবে ইউনিক নয়, কারণ একজন ছাত্র অনেক কোর্স নিতে পারে, আবার একটি কোর্সে অনেক ছাত্র থাকতে পারে। কিন্তু তাদের জুটি (Pair) সবসময় ইউনিক হতে হবে।

### ৪. At a Glance

| সুবিধা                                                        | সীমাবদ্ধতা                                                                               |
| :------------------------------------------------------------ | :--------------------------------------------------------------------------------------- |
| Data Integrity: ডুপ্লিকেট রিলেশনশিপ হওয়ার সুযোগ থাকে না।      | Complex Queries: জয়েন করার সময় একাধিক কলাম ম্যাপ করতে হয়, যা কোড বড় করে দেয়।             |
| No Extra Column: অযথা auto_increment_id কলামের প্রয়োজন হয় না। | Performance: ইনডেক্সের সাইজ বড় হয়ে যায়, যা অনেক বড় ডেটাসেটে পারফরম্যান্স স্লো করতে পারে। |

### ৫. কখন কম্পোজিট কি ব্যবহার করা উচিত?

- **Many-to-Many Tables:** দুটি টেবিলের মাঝখানে যখন কোনো 'Join Table' তৈরি করবেন তখন।
- **Weak Entities:** যেমন আমরা আগে আলোচনা করেছিলাম, যেখানে চাইল্ড টেবিল প্যারেন্টের আইডির ওপর নির্ভরশীল।
- **Mapping Tables:** যেখানে শুধু দুটি আইডির রিলেশনশিপ স্টোর করা হয়।

### নোটঃ

- **Don't over-stack:** ৩ বা ৪টির বেশি কলাম নিয়ে কম্পোজিট কি বানানো উচিত না। এতে কুয়েরি ম্যানেজ করা দুঃস্বপ্ন হয়ে দাঁড়াবে।
- **Alternative:** যদি রিলেশনশিপ খুব জটিল হয়, তবে একটি UUID বা ID কলাম (Surrogate Key) ব্যবহার করে কলামগুলোকে UNIQUE কনস্ট্রেইন্ট হিসেবে সেট করুন।
