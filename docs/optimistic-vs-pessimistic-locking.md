## Optimistic vs Pessimistic Locking: Choosing Your Strategy

লকিং (Locking) হলো কনকারেন্সি হ্যান্ডেল করার একটি ফিলোসফি। আপনার অ্যাপ্লিকেশন কি খুব বেশি ট্রাফিক হ্যান্ডেল করে যেখানে সংঘর্ষ (Conflict) হওয়ার সম্ভাবনা বেশি? নাকি এটি এমন একটি সিস্টেম যেখানে সবাই শান্তিতে নিজের মতো কাজ করে? এই প্রশ্নের উত্তরের ওপর ভিত্তি করেই আপনি বেছে নেবেন আপনার লকিং স্ট্র্যাটেজি।

যখন একাধিক ইউজার বা প্রসেস একই সময়ে একটি নির্দিষ্ট ডেটা আপডেট করতে চায়, তখন ডেটার সঠিকতা বজায় রাখার জন্য আমাদের লকিং মেকানিজম ব্যবহার করতে হয়। লকিং প্রধানত দুই প্রকারের হয়।

### ১. Pessimistic Locking (The "Trust No One" Approach)

**Pessimistic Locking** ধরে নেয় যে, "ডেটা কনফ্লিক্ট বা সংঘর্ষ অবশ্যই ঘটবে।" তাই এটি কোনো ডেটা রিড করার সময় বা আপডেট করার আগেই ওই রো-টিকে তালা মেরে দেয় (Lock), যাতে অন্য কেউ সেখানে হাত দিতে না পারে।

- **কিভাবে কাজ করে?** ইউজার-এ ডেটা রিড করার সময় একটি লক নেয়
  - ইউজার-বি যখন ওই ডেটা রিড বা আপডেট করতে আসে, তখন তাকে লাইনে দাঁড়িয়ে অপেক্ষা করতে হয় যতক্ষণ না ইউজার-এ তার কাজ শেষ করে লকটি রিলিজ (Commit/Rollback) করে।

**বাস্তব উদাহরণ:** একটি ব্যাংকের কাউন্টারে ক্যাশ জমা দেওয়া। আপনি যতক্ষণ কাউন্টারে কাজ করছেন, অন্য কেউ ওই নির্দিষ্ট কাউন্টারটি ব্যবহার করতে পারবে না।

### ২. Optimistic Locking (The "Hope for the Best" Approach)

**Optimistic Locking** ধরে নেয় যে, "সবাই ভদ্রভাবে কাজ করছে এবং কনফ্লিক্ট হওয়ার সম্ভাবনা খুব কম।" তাই এটি পড়ার সময় বা কাজ করার সময় কোনো তালা মারে না। এটি শুধু আপডেট করার ঠিক আগের মুহূর্তে চেক করে দেখে যে, অন্য কেউ এর মধ্যে ডেটাটি বদলে দিয়েছে কি না।

- **কিভাবে কাজ করে?** এটি সাধারণত একটি version বা timestamp কলাম ব্যবহার করে।
  - যখন আপনি ডেটা রিড করেন, তখন ভার্সনটিও রিড করেন (ধরুন, Version: 1)।
  - আপডেট করার সময় ডেটাবেজ চেক করে: WHERE id = ? AND version = 1।
  - যদি অন্য কেউ ইতোমধ্যে আপডেট করে ভার্সন ২ করে দেয়, তবে আপনার আপডেটটি ফেইল করবে। তখন আপনাকে নতুন ডেটা নিয়ে আবার চেষ্টা (Retry) করতে হবে।

- **বাস্তব উদাহরণ:** উইকিপিডিয়া বা গুগল ডক এডিট করা। একই সাথে অনেকে এডিট করতে পারে, কিন্তু সেভ করার সময় যদি কেউ আগে সেভ করে ফেলে, তবে আপনাকে কনফ্লিক্ট রেজোলিউশন করতে হয়।

### ৩. মূল পার্থক্যসমূহ (Head-to-Head)

| বৈশিষ্ট্য    | Pessimistic Locking                  | Optimistic Locking                  |
| ------------ | ------------------------------------ | ----------------------------------- |
| মূল চিন্তা   | দুর্ঘটনা ঘটবেই, তাই আগেই তালা মারুন। | দুর্ঘটনা কম ঘটে, তাই শেষে চেক করুন। |
| লক করার সময়  | ট্রানজেকশন শুরু বা রিড করার সময়।     | শুধুমাত্র আপডেট করার মূহুর্তে।      |
| পারফরম্যান্স | স্লো (অন্যদের ওয়েট করতে হয়)।         | ফাস্ট (কেউ কাউকে আটকে রাখে না)।     |
| ডেডলক রিস্ক  | অনেক বেশি।                           | একদম নেই বললেই চলে।                 |
| রিসোর্স খরচ  | ডাটাবেজ কানেকশন এবং মেমরি বেশি নেয়।  | খুব কম (শুধু একটি কলাম চেক করা)।    |

### ৪. কখন কোনটি ব্যবহার করবেন?

**Pessimistic Locking বেছে নিন যখন:**

- **হাই কনফ্লিক্ট:** একই ডেটা নিয়ে অনেকে মারামারি করে (যেমন- ফ্ল্যাশ সেলের স্টক বা ট্রেনের শেষ ৫টি সিট)।
- **রিট্রাই করা কঠিন:** যদি আপডেট ফেইল করলে আবার নতুন করে কাজ করাটা সিস্টেমের জন্য খুব ব্যয়বহুল হয়।
- **লং ট্রানজেকশন:** যদি ট্রানজেকশন অনেকক্ষণ ধরে চলে।

**Optimistic Locking বেছে নিন যখন:**

- **লো কনফ্লিক্ট:** বেশিরভাগ সময় ইউজাররা আলাদা আলাদা ডেটা নিয়ে কাজ করে (যেমন- ইউজার প্রোফাইল বা ব্লগ পোস্ট)।
- **স্কেলেবিলিটি জরুরি:** যখন হাজার হাজার ইউজার একসাথে থাকে এবং আপনি ডাটাবেজকে লকিং দিয়ে জ্যাম করতে চান না।
- **Disconnected Systems:** যেখানে রিড এবং রাইটের মাঝে অনেক বড় বিরতি থাকে (যেমন- একটি ওয়েব ফর্ম যা পূরণ করতে ইউজারের ৫ মিনিট লাগে)।

### ৫. কোড লেভেল ইমপ্লিমেন্টেশন (Concept)

**Pessimistic (SQL Example):**

```sql
BEGIN;
-- এই রো-টি এখন অন্য কারো জন্য লক হয়ে গেল
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- কাজ শেষ করুন
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

**Optimistic (Application Logic):**

```javascript
// Step 1: Read data with version
const user = await db.query("SELECT id, name, version FROM users WHERE id = 5");

// Step 2: Try to update ONLY if version is still the same
const result = await db.execute(
  "UPDATE users SET name = 'Nasim', version = version + 1 WHERE id = 5 AND version = ?",
  [user.version],
);

if (result.rowsAffected === 0) {
  throw new Error("Concurrency Error: Data was updated by someone else.");
}
```

### ৬. Performance Strategy

**১. Default to Optimistic:** আধুনিক স্কেলেবল সিস্টেমে আমরা সবসময় প্রথমে অপ্টিমিস্টিক লকিং দিয়ে শুরু করার চেষ্টা করি। এটি সিস্টেমকে অনেক বেশি হালকা রাখে।

**২. Deadlock Management:** পেসিমিস্টিক লকিং ব্যবহার করলে আপনার সিস্টেমে ডেডলক হওয়ার সম্ভাবনা বেড়ে যায়। তাই সবসময় লকিং অর্ডার ঠিক রাখুন।

**৩. Retry Policy:** অপ্টিমিস্টিক লকিং-এর জন্য একটি ভালো Backoff এবং Retry পলিসি রাখুন যাতে ফেইল করলে অটোমেটিক আবার চেষ্টা করে।

## নোটঃ

- **Don't Overuse FOR UPDATE:** অনেকে ভয় পেয়ে সব জায়গায় পেসিমিস্টিক লক দিয়ে দেয়। কিন্তু এটি অ্যাপ্লিকেশনের থ্রুপুট কমিয়ে দেয়।
- **Version Control:** আপনার প্রতিটি বড় টেবিলে একটি version বা updated_at কলাম রাখা একটি বেস্ট প্র্যাকটিস। এটি পরে অনেক কাজে লাগে।
